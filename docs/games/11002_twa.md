## Issue

TWA's rendering pipeline has three distinct problems:

### 1. AsyncFileIO:7 — ✅ FIXED
Directory enumeration now works. The handler follows the same async
request-object protocol as ordinal 3: reads [req+0x34/0x38] for callback
PC/data, enumerates subdirectories of `UserTrivia/Packs`, writes status
and entry count to [req+0x20/0x24], queues completion callback.

Returns entries like `["movies", "music", "sample1", "tv"]`.

### 2. Pack content loading — ❌ BLOCKED
After AsyncFileIO:7 callback returns pack names, the game calls
`AsyncFileIO:0` with `path="data"` — a generated index/cache file that
doesn't exist in the bundle. The game expects this file to contain
metadata about available packs. Without it, the game never opens the
actual .ipd/.tga files from the pack directory.

The "data" file is likely built by the game at runtime on a real iPod
from scanning pack directories. Our `AsyncFileIO:0` handler resolves
the path to `.clicky-saves/data` which creates an empty file.

### 3. Texture association — ❌ BLOCKED
The game uploads a 2×2 A8 placeholder texture (handle 0xDE1) then
binds material handle 0x28 for drawing 149×75 pack icons. The upload
and draw use completely different GL texture names — no upload matches
the draw's UV span.

Root cause: TWA's ordinal-45 (`glPrepareTexture`) creates empty
texture descriptors (all zeros at call time), then the game fills them
in later. Our ordinal-45 handler exits early because the descriptor
hasn't been populated yet. Even with the fix to not early-return,
the descriptor is still zeros — the game writes the texture name
AFTER ordinal-45 returns but BEFORE ordinal-99 uploads pixels.

The real pack icon textures would be loaded through the pack loading
pipeline (blocked by issue #2), so fixing the texture association
alone won't help until packs load.
